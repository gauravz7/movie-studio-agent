# MCP and Skills: the two standards behind agent integrations — for leaders, then builders

*I wrapped a real Google Cloud image model (nano-banana) as an MCP server, drove it from a Gemini
agent, hosted it, and layered Agent Skills on top. This post has two parts: **Part 1** is the
five-minute version for engineering leaders; **Part 2** is the full technical field report — code,
transcripts, and the token math.*

---

# Part 1 — For the CTO: the decision in five minutes

## The one idea
**MCP is USB-C for AI tools.** Write a capability once as an MCP server, and any AI app — Claude,
a Gemini agent, an IDE — uses it with no custom glue. It turns the **M×N** integration mess (every
app × every system) into **M+N** (build each capability once). That's the strategic shift:
integration stops being a per-app project and becomes a config change.

We measured it. Exposing nano-banana as an MCP tool was **~30 lines**; a Gemini agent adopted it in
**~5 lines**, zero glue, cross-vendor. The agent generated this by *calling a tool over a
protocol* — it never knew the image model existed:

![A banana astronaut, generated by a Gemini agent through MCP](images/agent-banana-astronaut.png)

## MCP vs Skills — you'll use both
There's a sibling standard: **Agent Skills** — a folder (`SKILL.md` + instructions + optional
scripts) an agent loads into its **own runtime**. It's an open spec (`agentskills.io`), supported
by **both Claude and Google ADK** (not Claude-only, as many assume).

> **MCP adds a *connection* — a capability the agent calls.
> A Skill adds *know-how* — a workflow the agent runs itself.** They compose.

| | Use it for | Runs | Credentials |
|---|---|---|---|
| **REST** | plain services, non-LLM consumers | your server | in your service |
| **MCP** | shared / external / credentialed capabilities, cross-vendor | your server | **centralized** |
| **Skill** | workflows & domain know-how | the agent's runtime | per-agent |

## The cost model (get this right)
AI spend has three parts, and conflating them is the classic mistake:

1. **The work the model does** — output tokens, or here the **per-image charge**. *Fixed by the
   task; no protocol or Skill makes it cheaper.* For a generation app, this dominates the bill.
2. **Input-token (context) cost** — the prompt you send every turn, *including every resident tool
   schema*. **MCP loads all tool schemas every turn; Skills load a one-line pointer and pull
   details on demand.** So Skills genuinely cut this — a real model-bill saving — and it's biggest
   for **text-heavy, many-tool, many-turn** agents.
3. **Hosting** — Skills need none; MCP is a service you run.

> **Skills cut hosting and input-token cost; only managing the work itself — quota, rate limits,
> model choice — cuts the dominant per-call spend.** Optimize the layer that's actually big for
> *your* workload.

## What you must govern
- **Consent** before any tool runs.
- **Two auth boundaries that never mix:** the user's token proves *who they are*; the server calls
  downstream APIs under its *own* identity — never pass the user's token through (the classic
  breach).
- **Untrusted metadata:** treat third-party MCP servers and Skills like third-party code.
- **Context = cost:** "connect everything" quietly inflates token spend; curate.

## Where it breaks at scale
At thousands of concurrent users the bottleneck is **model quota, not your servers.** Design for it:
**stateless** services for horizontal scale, **object storage** for artifacts, **per-user rate
limits** against a quota sized on purpose.

## Recommendations
1. **Adopt MCP** as the default contract for exposing systems/tools to AI; **adopt Skills** for
   packaging workflows and know-how.
2. Stand up a thin platform: a vetted registry of MCP servers + Skills, a standard deploy pattern
   (containerized, managed identity), and governance defaults.
3. **Budget for model consumption, not integration labor** — integration gets cheap; usage doesn't.
4. Prove it on one high-reuse capability, confirm the 30-lines-to-expose / 5-lines-to-adopt ratio,
   then scale the pattern.

*That's the decision. The rest is how it actually works.*

# Part 2 — For the builder: how it actually works

## MCP in one breath
MCP is a **wire standard** — JSON-RPC 2.0 with a fixed vocabulary. A **host** (the AI app) runs a
**client** that speaks the protocol to one **server** (your code). A server only ever sees its own
little conversation — never your whole chat or other servers. Isolation by design.

## Building the server: nano-banana as two tools
nano-banana (`gemini-3.1-flash-image`) does two things, so it becomes two tools. The flagship, with
three deliberate choices:

```python
@mcp.tool()
async def generate_image(prompt: str, ctx: Context, aspect_ratio: str = "16:9") -> GeneratedImage:
    await ctx.report_progress(0.1, 1.0, "contacting Vertex")     # (1) live progress
    resp = _client().models.generate_content(
        model="gemini-3.1-flash-image", contents=prompt)
    data, mime, _ = _extract_image(resp)
    path = _save_image_bytes(data, mime, stem="gen")
    return GeneratedImage(                                        # (2) structured output
        resource_uri=f"image://{path.name}",                     # (3) a link, not 2 MB
        size_bytes=len(data), model="gemini-3.1-flash-image", prompt=prompt)
```

1. **Progress** — image gen takes ~12s; without it the client just spins.
2. **Structured output** — a typed return gives callers clean JSON, not a string to re-parse.
3. **A link, not the bytes** — returning 2 MB of base64 would bloat context every call. (Same
   principle Skills use to save context, later.)

The edit tool is the same shape with a source image. I made a raccoon astronaut, then asked
`edit_image` to "give it a tiny top hat" — one surgical change, through the protocol:

![A raccoon astronaut wearing a tiny top hat, edited via MCP](images/raccoon-tophat-edited.png)

## The three server primitives — by *who's in control*
- **Tools** → the **model** calls them.
- **Resources** → the **app** reads them (file-like context by URI; mine has `config://app` and a
  templated `image://{name}` that reads a generated image back).
- **Prompts** → the **user** runs them (slash-command templates). **When they matter:** a human in a
  UI host (Claude Desktop, an IDE) wanting an expert-crafted, reusable starting point. Autonomous
  agents mostly bypass them (they drive via tools). A prompt is a tiny, user-triggered "skill over
  the wire" — instructions only.

## The twist: the server can call *back*
Think of your tool as a **baker making your cake**:
- **Progress** — texts you status ("20 min left"). No reply.
- **Elicitation** — needs info only you have ("what name on the cake?"). You answer.
- **Sampling** — borrows a brain: asks the *host's* LLM to think for it (my `caption_last_image`).

These are a **matched pair**: the server calls them, but only if the client registered the callback
*and* advertised the capability at handshake. My raw client supports all three; the Gemini ADK agent
supports progress and *optional* sampling, but **not** elicitation — so those tools degrade
gracefully there.

## Watching the raw protocol (no LLM in the loop)
A bare client, straight on the wire:
```
== INITIALIZE ==  server: learn-mcp 1.28.1   protocolVersion: 2025-11-25
add(2,3) -> 5.0
divide(1,0) isError -> True | Cannot divide by zero…
generate_image -> {'resource_uri': 'image://gen.png', 'size_bytes': 1985099}
elicitation -> user: robot barista, cyberpunk    sampling -> "One small step for fruit…"
```
Two lessons: the version negotiated to **`2025-11-25`** (trust the handshake, not a constant), and
`divide(1,0)` came back as a **success flagged `isError:true`** — not a crash.

> Golden rule: a *logic* failure returns `isError` so the model can read it and recover; only a
> *broken call* (unknown tool, bad args) is a true JSON-RPC error. The conversation keeps going.

## The payoff: a Gemini agent, zero glue
```python
Agent(name="mcp_image_agent", model="gemini-2.5-flash",
      tools=[McpToolset(connection_params=StreamableHTTPConnectionParams(
          url="http://localhost:9000/mcp", timeout=120))])   # ← that timeout matters
```
```
USER: Generate an image of a banana astronaut above the moon.
  [Gemini → MCP] generate_image({'prompt': 'a banana astronaut ...'})
  [MCP → Gemini] {'resource_uri': 'image://gen_3.png', 'size_bytes': 1837224}
  [Gemini]: Here's your image: image://gen_3.png
```
**War story:** my first image request through the agent failed *silently*. ADK's default MCP tool
timeout is **5s**; nano-banana takes ~12s. `timeout=120` fixed it. No tutorial mentions this.

## Transports and state, in plain terms
**Transport = the phone line.** *stdio* = the server runs on your machine over pipes (same room);
*Streamable HTTP* = it runs on the web over HTTPS (a phone call). "Streamable" means the server can
keep sending on the open line (SSE) — which carries progress and follow-ups. Switching my agent from
local to hosted was one line: `StdioConnectionParams` → `StreamableHTTPConnectionParams`.

**Stateful vs stateless = does the server remember you?** A **stateful** server is a *barista who
knows you* ("customer #42, the usual?") — it holds your session, and `#42` is the `Mcp-Session-Id`
the server issues at `initialize` and you echo each request. A **stateless** server is a *vending
machine* — no memory, no ticket, but you can clone a hundred. Stateful enables the
progress/sampling/elicitation trio; stateless scales trivially. MCP supports both.

## Hosting it
I deployed to Cloud Run. The concept that matters is the **two auth boundaries that never mix**: the
user's token proves *who they are* to the server; the server calls the model under its **own**
managed identity (no key file in the image). Never forward the user's token downstream.

## Skills: the other way to extend an agent
A Skill uses **progressive disclosure** — three levels loaded on demand: **L1 metadata** (always
resident, ~one line), **L2 instructions** (loaded when triggered), **L3 resources/scripts** (loaded
when referenced). I gave one ADK agent **both**: four Skills (poster/social/logo/edit) carrying
*workflows*, and the MCP image tools carrying *capability*. Asked for a jazz poster, it loaded the
`poster-designer` skill, read its L3 recipe, then called the MCP tool — and the recipe shaped the
output (note the **headline space at the top**, exactly as instructed):

![A rooftop jazz poster with headline space at the top](images/skill-mcp-jazz-poster.png)

```python
skills = [load_skill_from_dir(f"skills/{n}") for n in
          ("poster-designer","social-post","logo-maker","photo-editor")]
Agent(name="creative_studio", tools=[SkillToolset(skills=skills), mcp_tools])
```
With four skills installed, a logo request fired only `logo-maker`, an Instagram request only
`social-post`; the rest stayed at their ~44-token metadata. That's the context lever.

## The context-window economics (measured)
```
MCP — 9 tools, ALWAYS resident every turn: ~2,040 tokens
  (generate_image ~332, list_generated_images ~644, add ~107, …)
SKILLS — 4 skills: ~174 tokens resident (L1); L2 body (~110–294) loads only when a skill fires
```
A scaling law — MCP/REST are **O(N) always-on**; Skills are **O(N) tiny + O(1) on-demand**:

| Capabilities | REST / MCP (resident/turn) | Skills (metadata + 1 loaded) | Context saved |
|---|---|---|---|
| 5   | ~1,130  | ~380   | ~66% |
| 50  | ~11,300 | ~2,330 | ~79% |
| 100 | ~22,600 | ~4,500 | ~80% |

Since context tokens *are* input tokens, this is a real per-turn saving (~5× on capability-context;
~$68k vs $14k/mo illustrative at 1M sessions). **Caveats:** it doesn't touch the *work* cost (each
image is priced the same), and prompt caching narrows MCP's gap when enabled. *(Why MCP ≳ REST? MCP
auto-exposes the whole toolset and its schemas carry `outputSchema`/annotations REST omits — my
`add` is 107 tokens, `list_generated_images` 644. Those tokens buy discovery and typed I/O.)*

## What breaks at 1,000 concurrent clients
In order: **(1) model quota**, not compute — concurrent calls `429` and cost scales linearly; **(2)
stateful sessions vs scaling** — go stateless HTTP for request/response tools; **(3) the ephemeral
per-instance filesystem** — use object storage + signed URLs. The MCP server is never the hard part.

## Takeaways — what to actually do

**A. Choosing REST vs MCP vs Skills.** *REST = the app calls a service; MCP = the agent calls a
service; a Skill = the agent runs know-how itself; they compose.* Hardcode one-tool-one-app you own →
**MCP** for shared/external/credentialed capabilities → **Skills** for workflows & domain knowledge →
**combine** them in a capable agent (how vs what).

**B. Cost — three layers.** (1) *The work* (each image) — fixed; manage with quota/rate-limits/model
choice; dominates a generation app. (2) *Input-token/context* — Skills cut it via progressive
disclosure (~80% at scale); on MCP, filter tools, tighten schemas, return links, enable caching.
(3) *Hosting* — Skills none, MCP a service. **The rule:** Skills cut hosting + input-token cost; only
managing the work cuts the dominant per-call spend.

**C. Build & scale.** Return links not blobs; use structured output; handle tool failures as
`isError` data; set real timeouts (12s vs the 5s default); go stateless + object storage; the
bottleneck is model quota.

*The full server, clients, ADK agents, Skills, and a complete theory reference are in the repo. I set
out to understand MCP by building it — and walked away with a clear sense of when to use it, when to
reach for Skills instead, and how the two divide the work.*
